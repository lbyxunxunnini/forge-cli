## ② 代码能力

**验收标准：** 能处理复杂项目，借助结构化抽象能力

**内部优先级：** A 补全工具链 → C 项目扫描/抽象层 → B 任务拆分优化

**当前痛点：**
- Agent 不理解项目结构，生成的代码和现有架构不匹配
- 复杂任务一次性生成，质量差、容易出错
- 代码风格不一致
- 工具调用能力不足
- 项目内工具偏少
- 仅依赖文本编写，没有结构化代码操作

### A. 补全工具链

| 类别 | 工具 | 用途 | 状态 |

| 读代码 | AST 解析 | 理解代码结构 | 待做 |
| 读代码 | 符号搜索 | 按函数/类名定位 | 待做 |
| 读代码 | 依赖图 | 模块间关系 | 待做 |
| 读代码 | 引用查找 | 找到谁在用这个符号 | 待做 |
| 改代码 | 结构化编辑 | 改签名、重命名 | 待做 |
| 改代码 | Diff 生成 | 变更预览 | 待做 |
| 验证 | Lint / 类型检查 | 代码质量 | 待做 |
| 验证 | 测试执行 | 功能正确性 | 待做 |
| 项目管理 | Git 操作 | 版本控制 | 待做 |

## ③ 工作流抽取

**验收标准：** 先把 flutter-forge 抽成独立模式，支持快速接入新工作流

**核心概念：** meta-skill — 把"skill → 工作模式"的转换过程本身做成一个流程化的 skill

- [ ] 把 flutter-forge 从 skill 抽取为工作模式
- [ ] 制作 meta-skill（skill → 工作模式的转换流程）
- [ ] 提供模板项目，快速理解和接入新工作流

**输入：** 一个 skill 文件（如 SKILL.md）
**输出：** 自动在 flutter-forge-cli 中注册为一种工作模式，可通过命令切换

## ⑥ 上下文压缩

**验收标准：** 不丢失关键信息、自动摘要、token 预算管理、对齐 Claude

**当前痛点：**
- 长对话 token 超限，模型报错或截断
- 旧信息丢失，模型忘记之前说过的内容
- 上下文太长导致响应变慢、成本高
- 工具调用结果（如文件内容）占用大量 token

**计划：**
- [ ] 调研 Claude Code 的具体实现
- [ ] 自动摘要（对话过长时自动压缩早期消息）
- [ ] token 预算管理（动态控制上下文窗口）
- [ ] 工具结果裁剪（大文件内容只保留关键部分）
