# Question Budget

提问预算服务的是“当前阶段需要补多少关键输入”，不是“默认把几个角色都问一遍”。

## 预算按任务类型分配

| 任务类型 | 预算 | 原则 |
|----------|------|------|
| 轻量任务 | `1` 问 | 豁免最强写前确认等待；不制造额外确认，发现边界风险必须升级 |
| 中等任务 | `1-3` 问 | 只问会影响扫描方向或实现落点的问题 |
| UI 优化 | `1-3` 问 | 只问会影响视觉层级、布局结构或交互边界的问题 |
| 架构级任务 | `2-4` 问 | 只问会影响模块边界、状态归属、路由接入、迁移范围的问题 |
| 功能开发 / 页面开发 | `2-6` 问 | 先收需求边界，再补方案关键缺口 |
| 新项目共创 | `6-10` 问 | 必须分里程碑提问，不一次性问满 |

## 预算按规模修正

| 规模 | 修正规则 |
|------|----------|
| 小 | 尽量压到最低预算 |
| 中 | 使用标准预算 |
| 大 | 允许上浮，但必须分阶段提问 |

硬规则：

- `大任务` 不代表问题可以无限增加
- 问题数量越多，不代表流程越完整
- 每一问都必须服务当前阶段推进
- 多角色发现缺口时，问题先回传 `controller` 去重和排序，不按角色轮流各问一遍
- 对用户一轮只放 1 个最高优先级问题；用户回答后再继续下一问

## 各阶段怎么问

### S1 需求确认

只问会影响以下内容的问题：

- 页面目标
- 核心区块
- 返回路径
- 内容来源
- 关键交互

推荐格式：

```text
[f-forge] 需求分析师：
- 已知：...
- 待确认 1：...
- 待确认 2：...
```

### S2 方案确认

只问会影响以下内容的问题：

- 视觉层级
- 模块边界
- 组件复用
- 状态归属
- 路由接入

### 大改版 / 长文档场景

当输入为超长 PRD、超长 UI 文档或整体大改版时：

- 先压缩成当前阶段摘要包
- 只问会阻断当前阶段推进的问题
- 不把原始长文档里的所有未决项一次性全问出来
- 仍然遵守每轮只问 `1` 个关键问题的一问一答原则

### C0-C3 共创阶段

共创阶段必须分里程碑提问：

- `C0`：目标、用户、场景
- `C1`：页面结构、内容组织、风格方向
- `C2`：目录结构、状态管理、路由、网络层
- `C3`：首批页面或模块范围

## 总规则

- 用户已经明确提供的信息，不重复追问
- 能安全采用默认值时，先声明默认值，不必强追问
- 用户只说“确认 / 可以 / 按这个来”时，只能确认上一轮摘要中的明确条目
- 没有进入对应阶段时，不提前问下游问题
