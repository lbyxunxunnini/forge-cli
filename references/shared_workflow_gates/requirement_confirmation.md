# Requirement Confirmation

通用需求确认门禁：

1. 需求分析结束后，不允许默认直接切到下一个角色
2. 只有当 `用户确认状态 = 允许进入下一角色` 时，才允许进入 UI / 架构 / 实现
3. 只要缺口会影响页面形态、信息架构、内容来源、路由、返回路径、首屏目标、分享/埋点/搜索等页面能力、核心交互，就必须先问用户
4. 如果 `需求分析师`、`UI 设计师`、`架构设计师` 同时发现缺口，必须先由 `controller` 去重和排序，再交给对应角色发出当前这一问
5. 大改版或长文档场景下，先输出当前阶段摘要包，再提问题
6. 问题必须贴着当前阶段，不把后续阶段的未决项提前一次性问完
7. 对用户每轮只问 1 个最高优先级问题，收到回答后再进入下一问
8. 需求分析输出应控制为：
   - 3-5 行已知摘要
   - 当前最高优先级的 1 个待确认问题
9. 没有拿到回答前，不要输出完整方案，不要切下一个角色
10. 用户只回复“确认 / 可以 / 按这个来 / 没问题”时，只能确认上一轮明确摘要中的条目
11. 如果仍有摘要外的未决项，必须先回写“已确认 / 待确认”，不能把一句确认扩展成全部通过

提问预算补充：

- `L1`：0-2 问，通常不进入完整确认门禁
- `L2`：2-4 问，确认项只保留会改变页面结构或交互的内容
- `L3`：4-7 问，允许覆盖流程、异常、边界和复用取舍
- `L4`：6-10 问，但必须分阶段确认，不一次问完

确认门禁追求的是“够用收口”，不是“问题越多越安全”。

通用核心页面最低确认项：

- 页面目标
- 核心信息和区块
- 内容来源
- 用户路径
- 关键交互
- 边界状态
- 页面能力（分享 / 搜索 / 筛选 / 分页 / 埋点 / 深链等）
