# Flutter Forge Reference - 新项目共创模式

`新项目共创` 是 `controller` 主导的前代码阶段轨道，不是“先把四个角色都跑一遍”。

## 目标

把模糊想法推进成可以进入执行轨道的第一版项目范围。

共创阶段最终应产出：

1. 项目确认摘要
2. 页面 / 模块结构草案
3. 风格方向
4. 工程组织方向
5. 首批范围冻结结果

## 轨道与阶段

新项目共创固定走：

- `C0` 想法收口
- `C1` 方向共创
- `C2` 工程定型
- `C3` 首批范围冻结

只有 `C3` 完成后，才允许进入执行轨道 `S3`。

## 各阶段职责

### C0 想法收口

由 `controller` 主导，对外以 `[f-forge] 主控：xxx` 标签发问（C0 尚未进入 S1，需求分析师未启动，详见 [decision_and_question_protocol.md](decision_and_question_protocol.md) 第 2 节例外条款）。

目标：

- 明确项目想解决什么问题
- 明确目标用户
- 明确核心场景

此阶段：

- 不进入实现
- 不讨论完整文件结构
- 不展开完整 UI 包

### C1 方向共创

按需调用 `ui-agent`。

目标：

- 收口页面结构方向
- 收口信息组织方式
- 收口风格方向

此阶段输出“方向和结构草案”，不输出完整页面包。

### C2 工程定型

按需调用 `arch-agent`。

目标：

- 冻结目录结构方向
- 冻结状态管理方向
- 冻结路由和网络层方向
- 决定guardrails策略

此阶段可以初始化或补齐guardrails，但仍不进入实现。

### C3 首批范围冻结

由 `controller` 主导。

目标：

- 明确第一版包含哪些页面 / 模块
- 明确哪些不进入第一版
- 明确可以进入执行轨道

通过后，从 `C3` 切到 `S3`。

## 闸门规则

### C0 -> C1

必须至少明确：

- 项目目标
- 目标用户
- 核心场景

### C1 -> C2

必须至少明确：

- 页面结构方向
- 主页面核心区块
- 一条默认视觉方向

### C2 -> C3

必须至少明确：

- 目录组织方向
- 状态管理方向
- 路由组织方向
- guardrails策略

### C3 -> S3

必须至少明确：

- 首批页面 / 模块范围
- 第一版实现边界
- 当前已允许进入代码阶段

## C3 → S3 等价性说明

共创轨道 `C0-C3` 的产出覆盖了执行轨道 `S1 + S2` 的放行条件：

| 共创阶段 | 等价的执行阶段 | 产出 |
|----------|---------------|------|
| C0-C1 | S1 需求确认 | 项目目标、核心场景、页面结构方向 |
| C2-C3 | S2 方案确认 | 工程组织、状态管理、路由、首批范围冻结 |

因此 `C3` 完成后直接进入 `S3`，不需要重复走 `S1` 和 `S2`。

如果 `C3` 完成后首批范围足够小（1-2 个页面，无需拆包），可以跳过 `S3` 直接进入 `S4 实现中`。

## 用户可见输出

共创阶段的可见输出应该是阶段状态，不是角色排队：

```text
[f-forge] 模式：新项目共创
- 升级原因：当前只有产品方向，需要先收口页面结构和工程方案
[f-forge] 阶段：C1 方向共创
[f-forge] UI 设计师：建议先确定首页、列表页和详情页的结构草案。
```

## 不要做的事

- 不要在 `C0-C2` 直接进入实现
- 不要把一句“确认”扩展成跨阶段放行
- 不要一次性把需求、UI、架构、实现整包倾倒给用户
- 不要把共创模式重新写回旧的角色顺序主流程
