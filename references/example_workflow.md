# Flutter Forge Reference - Example Workflow

这个文件展示一个真实任务在现行协议下的推荐流转方式。它说明的是：

- 主控如何推进阶段
- 哪些专项判断会参与
- 如何从收口走到实现

## 场景

用户给出：

- 一份会员中心页面 PRD
- 一张会员中心页面设计图
- 一个现有迭代中 Flutter 项目

目标：

- 在已有项目中新增“会员中心”页面
- 尽量复用已有模式
- 先收口，再实现

## 推荐流程

### 启动握手（仅在命中条件时）

```text
[f-forge] 模式：启动握手
- 项目阶段：迭代项目
- guardrails：未发现
- Flutter skills：已就绪
- 下一步：进入页面开发
```

### 进入任务

```text
[f-forge] 模式：页面开发
- 升级原因：新增页面，涉及页面结构、状态接入和路由组织
[f-forge] 阶段：S1 需求确认
```

### S1 需求确认

```text
[f-forge] 需求分析师：
- 已知：会员中心需要展示会员状态、权益入口和续费入口
- 约束：必须覆盖未开通、已开通、过期、加载中、失败态
- 待确认 1：未登录用户是否允许进入
- 待确认 2：过期态是否保留权益说明区
- 结论：待确认后进入方案确认
```

### S2 方案确认

```text
[f-forge] 阶段：S2 方案确认
[f-forge] UI 设计师：
- 来源：真实视觉输入
- 结构：顶部信息区、会员状态卡、权益区、底部主操作区
- 待确认：空态和失败态布局未明确
- 结论：结构足够进入架构判断

[f-forge] 架构设计师：
- 结构决策：落在 lib/features/member_center/
- 冻结约束：沿用页面级状态方案，不直接修改现有公共会员卡
- 复用策略：复用 shared loading/error scaffold，权益列表项暂不抽公共
- 结论：允许进入实现
```

如果当前没有足够的设计图、参考图或文字化 UI 描述，`S2` 不应直接放行，应先补要输入：

```text
[f-forge] 阶段：S2 方案确认
[f-forge] UI 设计师：
- 来源：需求文字描述
- 待补输入：缺少会影响布局放行的设计图或参考图
- 结论：先向用户补要 UI 输入，再继续冻结结构
```

### S4 实现中

```text
[f-forge] 阶段：S4 实现中
[f-forge] 页面工程师：
- 当前动作：生成页面结构、状态接入骨架和 loading/error 占位
- 实现范围：member_center_page.dart、相关 widgets、页面状态文件
- 风险：会员等级映射和按钮跳转仍需按真实接口补齐
- 验证：完成基本自检
```

### S5 验证中

```text
[f-forge] 阶段：S5 验证中
[f-forge] 页面工程师：
- verify-agent 验证完成：已覆盖新增页面文件、状态接入和基础结构一致性
- 结果：结构和命名对齐当前项目主流
- 风险：真实接口字段映射待接入
- 结论：可进入完成，保留接口待补项
```

### S6 完成

```text
[f-forge] 阶段：S6 完成
[f-forge] 本轮完成：已完成会员中心页面结构、状态接入和基础验证；修改了页面骨架、目录落点和复用边界；实现了新增页面的首轮收口。
[f-forge] 本轮完成：任务已完成，是否需要更新guardrails
- 是：更新guardrails
任务结束，已自动退出 flutter-forge
```

## 这个示例想说明什么

1. 主控先推进阶段，不是先把角色全部打出来
2. 需求、UI、架构、实现只在需要时介入
3. 验证是独立收口阶段，不是实现顺手带过
4. 大任务结束前先给结果摘要，再显式判断一次是否需要更新guardrails

## 模式补充

这个示例展示的是“新增会员中心页面”，所以对外模式是：

```text
[f-forge] 模式：页面开发
```

如果同样是大任务，但核心不是“新页面/新模块”，而是已有项目中的业务闭环、跨页面状态联动、弹窗/提示栏/流程承接，则应改为：

```text
[f-forge] 模式：功能开发
- 升级原因：需求涉及业务闭环和跨页面状态联动，不以新增页面为核心
[f-forge] 阶段：S1 需求确认
```
