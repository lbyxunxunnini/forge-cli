# Forge CLI Reference - Engineering Heuristics

这个文件补充可执行判断标准，避免角色只停留在抽象职责。

## 页面拆分原则

页面拆分优先级：

1. 先按业务区块拆
2. 再按交互边界拆
3. 最后才按纯视觉块拆

“拆得自然”通常表示：

- 每个区块能用一句业务语义描述
- 区块内部状态变化比区块之间更内聚
- 区块边界不会频繁跨层共享局部状态
- 单个组件不会同时承担多个明显不同的业务职责

不自然的拆法通常表现为：

- 一个组件名很泛，例如 `ContentWidget`
- 一个组件内部混了标题区、列表区、底部操作区，却没有清晰边界
- 一个组件既负责页面布局，又负责大量业务状态判断

## 公共组件抽取标准

默认不要因为”看起来像”就抽公共组件。

满足以下任一条件才值得抽公共（与 architecture_designer.md 一致）：

1. 复用次数：已有 2+ 处相似实现
2. 修改频率：改动时需要同步改多个地方
3. 一致性要求：多个入口必须保持一致
4. 复杂度：组件逻辑复杂，抽取后能降低单页面的认知负担

出现以下情况时，不要急着抽公共：

- 只在一个页面里用到
- 第二个使用场景只是可能会来，不是已经存在
- 为了复用而加很多布尔参数
- 抽出来后命名变得空泛

### 复用追踪

相似实现检索会自动记录复用模式到guardrails的 `reuse_knowledge.reusable_patterns`。当同一模式出现第 2 次时，主动建议抽取公共组件。详见主文档 `SKILL.md` 的”复用追踪与公共组件建议”。

## 主流模式识别标准

判断”项目主流模式”时，不要只看单个文件。

建议顺序：

1. 先搜索相似实现（详见主文档 `SKILL.md` 的”相似实现检索”章节）
2. 看最近 3 到 5 个代表性页面
3. 看数量上最常见的写法
4. 排除明显历史遗留和孤例
5. 如果最近写法和全局旧写法冲突，优先以最近主流为准，但标记低置信度

### 低置信度判定标准

满足以下任一条件时，判断应标记为低置信度：

- 仅基于 1-2 个文件推断，样本不足
- 与项目中多数写法矛盾
- 存在多种合理解释，无法确定唯一主流
- 依赖的文件是历史遗留（修改时间早于项目主流技术栈切换时间）
- 判断涉及跨模块边界，但只看了单模块代码

低置信度判断的处理：标注 `confidence: low`，在guardrails中记录为参考项而非强制约束，并在首次使用时告知用户。

适合判断为主流的对象：

- 目录结构
- 页面命名
- 状态管理接法
- 共享组件边界
- API 请求入口

## 项目专项规则

### BuildContext

- 不要在异步 gap 之后盲目使用旧 `BuildContext`
- 不要把 `BuildContext` 长期存起来作为状态字段
- 需要依赖上下文时，优先在同步作用域内消费

### Widget / Element / RenderObject 认知

- 页面拆分和复用判断主要围绕 Widget 层语义进行
- 性能和布局问题排查时，要意识到 Widget 配置、Element 生命周期和 RenderObject 布局阶段不是一回事
- 遇到布局溢出、约束异常、重建异常时，要优先检查约束链和 rebuild 边界

### const 使用

- 稳定且无运行时变化的子组件优先考虑 `const`
- 不要为了机械加 `const` 而破坏可读性
- 如果组件频繁重建但配置稳定，`const` 是低成本优化点

### StatefulWidget 生命周期

- 本地短生命周期交互状态才放页面内部状态
- 依赖外部数据流或跨组件共享的状态，不要硬塞进 `StatefulWidget`
- 涉及订阅、controller、focus node、animation controller 时，必须考虑初始化和释放

### 状态管理选型

- 页面私有的短状态：局部 state
- 页面级业务状态：沿用项目主流方案，如 Provider / Riverpod / Bloc / Controller
- 跨页面共享状态：只在确有共享需求时进入更高层

选型时优先：

1. 先服从项目主流
2. 再考虑当前复杂度
3. 最后才考虑理论最优

### Sliver 使用场景

以下情况优先考虑 Sliver：

- 页面有可折叠头部
- 多段可滚动内容需要拼接
- 需要复杂滚动联动

简单单列表或简单表单页面，不要过度引入 Sliver 复杂度。

### 测试与预览

如果项目已安装官方 项目 skills，以下情况优先委托：

- 组件交互验证：`add-widget-test`
- 新组件展示和状态对照：`add-widget-preview`
- 真实页面流程验证：`add-integration-test`
