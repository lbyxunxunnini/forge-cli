# Forge CLI Reference - Fast Mode

`ff-fast` 是快速执行策略，不是独立任务模式。它的目标是提效：轻量优先、自动生成项目摘要、减少非关键前置流程，除非发现明确架构风险才升级到完整流程。未升级的轻量路径豁免最强写前确认等待；升级到中等及以上后按对应模式门禁执行。`ff-fast` 只减少读取和确认成本，不减少真实性门禁。

## 触发方式

满足任一时启用快速策略：

- 用户输入以 `ff-fast` 开头
- 用户明确说“快速处理”“轻量优先”“先快速定位，必要时再升级”

启用后仍按 `SKILL.md` 的任务路由判定真实模式，但默认倾向：

- 能轻量做就走轻量
- 需要局部扫描就走中等
- 只有出现需求缺口、UI 结构决策或架构边界风险时才升级

## 核心行为

1. 启动时优先生成 `project_snapshot`，作为冷启动摘要。
2. 轻量任务最多读 1-3 个关键文件。
3. 轻量任务不强制输出写前改动契约，保持快速路径。
4. 中等任务执行“15 分钟内完成策略”：扫描相似实现 → 改动契约 → 用户确认 → 最小改动 → 验证。
5. 只有明确风险才升级，不因为“新增页面”四个字机械进入重流程。

## 快速策略不等于全自动

`ff-fast` 可以减少确认，但不能替代中等及以上任务的用户决策：

- 未升级轻量路径可直接执行；中等及以上不自动确认功能、逻辑或 UI 改动契约。
- 不自动补目标、范围、验收、约束。
- 不自动冻结当前子单元。
- 不自动拍板高风险架构迁移。
- 不绕过安全、生产、支付、隐私、密钥相关确认。
- 不跳过验证。
- 不绕过工作模式锁和退出许可。
- 不假装读过不存在的guardrails。

## 自动扫描摘要

`ff-fast` 启动时可调用：

```bash
python3 scripts/project_snapshot.py /path/to/project
```

摘要至少包含：

- 目录结构概览
- pubspec 技术栈
- 路由入口
- 状态管理入口
- 网络层入口
- 测试入口
- guardrails路径状态

如果存在guardrails的 `quick_context`，优先使用；如果过期或缺失，再重新扫描。

## 升级条件

遇到以下情况必须升级（可操作判定信号）：

- 任务目标不清，需要询问用户超过 1 个问题才能继续。
- UI 结构会影响组件边界或交互路径。
- 需要新增 / 调整路由、状态管理、网络层或公共组件边界。
- 必须改共享核心文件且 write scope 不清晰。
- 简化/抽取/复用会影响业务链路、弹窗触发时机、共享工具入口或兜底分支。
- 发现项目guardrails与实际代码冲突。

## UI 信息不足时的边界

`ff-fast` 遇到 UI 信息不足时，按以下规则处理：

- **不影响实现正确性**（如纯文案修改、颜色调整）：保持轻量路径，豁免最强写前确认等待。
- **影响组件边界或交互路径**（如新增区块、调整布局结构）：必须升级到页面开发或 UI 优化，从 S2 开始。不为速度硬做。
- **影响业务逻辑表达**（如空态、错误态、加载态缺失）：升级到功能开发或页面开发，从 S1 开始。

`ff-fast` 不会在 UI 信息不足时自动采用推荐草案——那是 `ff-a` 的行为。

升级输出（先发模式切换日志，再发角色发言）：

```text
[f-forge] 模式：页面开发
- 升级原因：ff-fast 发现架构边界风险（新增页面需路由接入和状态归属判断），已超出轻量 / 中等范围
[f-forge] 阶段：S2 方案确认
[f-forge] 页面工程师：回到 S2 方案确认，先冻结组件边界和状态归属。
```

升级本质是模式切换，必须先输出 `[f-forge] 模式：xxx` 配合升级原因，才能输出角色发言。仅输出角色行不符合 P0 模式日志规范。

## 升级后降级

升级到重流程后，如果扫描发现实际复杂度低于预期（如需求已明确、UI 结构简单、无架构风险），可降级回中等任务：

```text
[f-forge] 页面工程师：扫描后发现需求已明确且无架构风险，降级为中等任务继续执行。
- 降级原因：xxx
```

降级必须有具体原因，不能无理由跳回轻量。

## 可见输出

启动：

```text
[f-forge] 页面工程师：ff-fast 快速策略，轻量优先，发现架构风险再升级。
```

中等任务：

```text
[f-forge] 页面工程师：ff-fast 中等任务，先用 project snapshot 和 1-3 个相似实现定位改动点。
[f-forge] 页面工程师：改动契约：允许改动订单筛选条件渲染；禁止改动路由、状态管理和接口协议；行为不变项：已有筛选结果与埋点保持一致；确认状态：等待用户确认。
```

完成：

```text
[f-forge] 页面工程师：已按 ff-fast 完成修改并完成最小验证。
```

## 不要做的事

- 不要把 `ff-fast` 当成全自动重流程。
- 不要因为用户说快就跳过必要验证。
- 不要因为用户说快就把未确认目标、范围、验收、约束当成默认已确认。
- 不要在中等及以上任务里因为用户说快就跳过功能/逻辑/UI 改动契约确认。
- 不要在发现结构风险后继续硬做轻量。
- 不要在轻量任务中输出完整阶段日志。
- 不要在当前子单元未验证通过前切换到下一个子单元或提前收口。
