# Flutter Forge Reference - Autonomous Mode

`ff-a` 是全自动执行策略，不是独立任务模式。它的作用是减少确认轮次：缺少信息或改动边界需要收口时，由 controller 选择推荐方案继续推进，并在输出里明确标注采用了什么默认方案。它豁免最强写前确认等待，但不能绕过安全、不可逆、生产环境或全项目级架构切换等高风险中断，也不能自动补核心任务定义。

## 触发方式

满足任一时启用全自动策略：

- 用户输入以 `ff-a` 开头
- 用户输入以 `ff a` 开头
- 用户明确说“全自动”“自动做完”“不要反复确认”“缺少部分用推荐方案”

启用后仍按 `SKILL.md` 的任务路由判定真实模式，例如：

- `ff-a 新建商品详情页` → 全自动策略 + 页面开发
- `ff-a 做登录流程联动` → 全自动策略 + 功能开发
- `ff-a 做包体积优化` → 全自动策略 + 架构级任务

## 与 ff-fast 的区别

- `ff-a`：缺口和改动边界自动采用推荐方案；功能/逻辑/UI 写改动不强制等待用户确认。
- `ff-fast`：轻量优先，尽量少走流程；发现结构风险就升级。

## 核心行为

1. 不为普通非写入缺口中断用户。
2. 需求、UI、架构缺口先转为“推荐方案”。
3. controller 将推荐方案标记为 `auto_assumption`。
4. 下游阶段可以把 `auto_assumption` 当作本轮冻结输入继续执行。
5. 结果日志必须汇总“自动采用了哪些方案”。
6. 功能/逻辑/UI 写改动前可用 `auto_assumption` 自动冻结改动边界；只有命中高风险中断条件时才等待用户确认。

## 自动采用推荐方案的范围

可以自动决策非写入缺口：

- 页面空态 / 加载态 / 错误态的常规表现
- 列表分页、刷新、筛选的默认交互
- 常见按钮位置、卡片间距、标题层级
- 新页面文件命名和目录落点（遵守guardrails或现有相似实现）
- 状态管理接入建议（遵守项目主流；全项目级切换仍需中断确认）
- 路由接入建议（遵守项目主流；全项目级切换仍需中断确认）
- 测试补充范围（按任务强度选择最小必要测试）
- 未给设计图时的基础业务风格 UI 草案

以下内容不属于自动采用范围，必须保持“未确认”直到用户或上游阶段明确收口：

- 任务目标
- 任务范围
- 验收标准
- 关键约束
- 当前子单元的完成定义

## 必须中断确认的情况

即使启用 `ff-a`，以下情况也不能自动拍板：

- 会删除数据、覆盖用户文件、迁移不可逆数据结构
- 需要真实密钥、账号、支付、隐私权限或生产环境操作
- 状态管理、路由体系、目录架构发生全项目级切换
- 需要修改大量共享核心文件但 write scope 不清晰
- 需求存在互斥目标，且任一选择都会明显改变产品行为
- 自动方案会绕过安全、隐私、合规或宿主权限边界
- 自动方案会改变弹窗、授权、邀请、支付、分享等用户可见链路的触发时机、兜底分支或共享工具入口

中断时只问 1 个最高风险问题，并说明为什么不能自动继续。

## 阶段门禁变化

`ff-a` 不取消阶段门禁。它改变缺口和写前边界的“确认”来源：

- 普通模式：用户确认后进入下一阶段
- `ff-a`：controller 生成推荐方案并标记 `auto_assumption` 后进入下一阶段
- 功能/逻辑/UI 写改动：可由 controller 生成自动改动边界并标记 `auto_assumption` 后进入写操作；命中高风险中断条件时仍必须用户确认

因此：

- S1 仍要产出需求冻结摘要，但自动假设只能补非核心缺口，不能替代目标/范围/验收/约束确认
- S2 仍要产出 UI / 架构冻结摘要，但可以由推荐方案冻结
- S4 仍只能在 write scope 清晰后实现；`ff-a` 下 write scope 可由自动假设冻结，高风险事项除外
- S5 仍必须验证
- 当前子单元未验证通过前，`ff-a` 也不得切换到下一个子单元
- 未满足退出条件前，`ff-a` 也不得离开当前工作模式

## 输出格式

启动时先输出：

```text
[f-forge] 全自动：已启用 ff-a，非阻塞缺口将采用推荐方案推进；安全、不可逆或高风险架构决策才中断确认。
```

结束时补充：

```text
[f-forge] 全自动摘要：本轮自动采用 3 项推荐方案：空态文案、卡片列表风格、Riverpod 页面级状态接入。
```
