## 二、核心工具差距（P0）

### 1. 文件搜索工具

| 工具 | 说明 | 优先级 | 工作量 |
|------|------|--------|--------|
| **Glob** | 按模式搜索文件（如 `**/*.dart`） | P0 | 1天 |
| **Grep** | 按内容搜索（正则匹配） | P0 | 1天 |
| **LS** | 列出目录结构 | P0 | 0.5天 |

**为什么必须**：没有这些，Agent 无法自动探索项目，用户必须手动告诉 Agent 文件在哪里。

### 2. 代码理解工具

| 工具 | 说明 | 优先级 | 工作量 |
|------|------|--------|--------|
| **AST 解析** | 解析代码结构 | P0 | 3天 |
| **符号表** | 全局符号索引 | P1 | 2天 |
| **调用图** | 函数调用关系 | P2 | 3天 |

### 3. 网络工具

| 工具 | 说明 | 优先级 | 工作量 |
|------|------|--------|--------|
| **WebFetch** | 获取网页内容 | P1 | 2天 |
| **WebSearch** | 网络搜索 | P1 | 2天 |

### 工具生态对比

| 工具 | Claude Code | Flutter Forge |
|------|-------------|---------------|
| `Glob` | 文件模式匹配 | ❌ 缺失 |
| `Grep` | 内容搜索 (ripgrep) | ❌ 缺失 |
| `LS` | 目录列表 | ❌ 缺失 |
| `Read` | 文件读取（支持图片/PDF） | ✅ 有 readFile |
| `NotebookRead` | Jupyter notebook 读取 | ❌ 缺失 |
| `WebFetch` | 网页内容获取 | ❌ 缺失 |
| `WebSearch` | 网络搜索 | ❌ 缺失 |
| `TodoWrite` | 任务管理 | ❌ 缺失 |
| `Edit` | 精确编辑 | ✅ 有 editFile |
| `Write` | 文件写入 | ✅ 有 writeFile |
| `Bash` | 命令执行 | ✅ 有 runCommand |
| `Agent` | 子 Agent 启动 | ✅ 有 AgentPool |

## 三、上下文管理差距（P0）

### 4. 记忆系统

| 组件 | 说明 | 优先级 | 工作量 |
|------|------|--------|--------|
| **MEMORY.md** | 记忆索引文件 | P0 | 1天 |
| **用户记忆** | 用户角色、偏好 | P0 | 1天 |
| **项目记忆** | 项目结构、模式 | P1 | 2天 |
| **反馈记忆** | 纠正、确认记录 | P1 | 1天 |

**记忆架构**：

```
L1: 工作记忆 (当前上下文)    → 当前对话消息、临时变量
L2: 短期记忆 (Session)      → Session 持久化、阶段历史、决策记录
L3: 长期记忆 (跨 Session)   → 用户偏好、项目知识、错误经验
L4: 永久记忆 (配置)         → 系统配置、角色定义、工作流规则
```

**文件结构**：
```
~/.flutter-forge/memory/
├── MEMORY.md              # 索引（自动加载）
├── user_role.md           # 用户角色
├── user_preferences.md    # 用户偏好
├── project_structure.md   # 项目结构
├── feedback_testing.md    # 测试反馈
└── references.md          # 参考资料
```

### 5. 上下文压缩

当前实现：简单截断（删除最旧消息），会丢失重要上下文。

需要实现：
- **重要性标记**：标记消息重要性（P0）
- **选择性保留**：保留重要消息（P0）
- **摘要生成**：生成历史摘要（P1）
- **滑动窗口**：固定窗口大小（P2）

## 四、状态机差距（P1）

### 当前问题

1. **状态转换无验证**：任意状态可以跳到任意状态
2. **无状态历史**：不知道从哪来的
3. **无状态恢复**：崩溃后无法精确恢复
4. **状态和阶段脱节**：SessionState 和 Phase 是两套系统

### 需要实现

- 严格的状态转换验证
- 完整的状态历史记录
- 精确的状态恢复机制
- 状态和阶段统一

## 八、无法弥补的差距

1. **LLM 模型能力** - Claude Opus/Sonnet 是闭源顶级模型，取决于模型提供商
2. **Anthropic 官方支持** - Claude Code 有专门团队维护
3. **生态系统规模** - 社区、市场、插件数量

## 九、结论

**建议策略**：
1. 不要试图完全复制 Claude Code — 模型能力差距无法弥补
2. 聚焦差异化 — Flutter 专项优化、中文支持、国产模型
3. 渐进式发展 — 先达到可用，再追求完美
4. 社区驱动 — 建立插件生态，让社区贡献

**乐观估计**：按 Phase 1-2 完成后可达到 Claude Code 70% 的核心体验。
